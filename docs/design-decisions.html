Pingaro Design Decisions

Decision log for Pingaro, a Windows app for long term network quality monitoring. This document captures the product, UX, measurement, build, and release choices made during development.

Purpose Identity Scope Measurement Use Profiles Overall Status Visual Design Defaults Build History Maintenance Timeline

Product Purpose

Pingaro exists to answer a practical question: is the connection good enough for what the user is trying to do, and when did it stop being good enough?

  • Serve experts and non-experts: network experts get direct visualizations of response time, packet loss, p95 latency, and jitter. Non-experts can choose one or more use profiles and let Pingaro classify issues for them.
  • Focus on quality over time: the tool is not a one-shot speed test. It is meant to run while the user works, plays, or calls, then summarize what happened over time.
  • Plain language first: the UI should not force users to interpret every chart. The overall status label uses direct sentences such as "So far, the connection looks good."

Identity

Decision Outcome Reason
Name Pingaro Short, memorable, and directly connected to ping/network monitoring without using the taken name "goping".
Tagline Long term network quality monitor States the product category plainly and distinguishes it from quick speed tests.
Executable pingaro.exe One product, one app, one executable.
Icon assets/pingaro.ico embedded through rsrc.syso The exe file icon and the top-left GUI window icon should both show the Pingaro icon.
User-visible names Names for files, controls, metrics, releases, and other visible concepts should describe what they are used for. Good naming is part of the user experience. A misleading or overly generic name creates avoidable confusion even when the underlying feature works.

Application Scope

  • GUI only: the TUI version was removed. Pingaro now has only the graphical Windows application.
  • No terminal dependency: the app must launch without opening a terminal window that needs to stay open. Release builds use -ldflags="-H=windowsgui".
  • Native Windows UI: the UI uses the Go Walk toolkit for a native Windows desktop application rather than a web app or terminal interface.
  • Public GitHub distribution: the repository is public, and releases attach the built pingaro.exe asset with user-facing download and run instructions.

Measurement Model

Metric How Pingaro Presents It Decision
RTT Realtime chart in milliseconds. Keep a live view so users can see immediate spikes and loss events.
p95 RTT Aggregate chart per configured period. Use p95 rather than average so sustained high latency is visible without letting one extreme point dominate.
Loss Aggregate packet-loss percentage. Show loss separately because a connection can have acceptable RTT but still fail calls or games through drops.
One-way jitter Estimated as half the two-way jitter and shown only for relevant use profiles. Show jitter when it matters for audio/video, but keep the UI simpler when the selected uses do not need it.

Target Handling

  • Users can configure up to three target groups.
  • For Internet quality, Pingaro can ping multiple well-known hosts in parallel.
  • If at least one Internet target replies in a batch, the batch is treated as successful and the minimum RTT is used.
  • This reduces false alarms caused by one public host having a temporary issue.

Internet Use Profiles

Internet Use is a multi-select list. The selected profiles determine the thresholds for classifying good, minor, noticeable, and serious issues.

Profile Default RTT thresholds (ms) Loss thresholds (%) Jitter thresholds (ms) Shows jitter graph
email & browsing Yes 250 / 500 / 1000 3 / 8 / 15 80 / 150 / 300 No
remote desktop No 120 / 200 / 350 1 / 3 / 8 30 / 60 / 120 No
audio calls Yes 120 / 180 / 300 1 / 3 / 6 20 / 40 / 80 Yes
video calls Yes 150 / 250 / 400 2 / 5 / 10 30 / 60 / 120 Yes
online gaming Yes 80 / 140 / 220 0.5 / 1.5 / 4 15 / 30 / 60 No
Superhuman Gaming No 40 / 80 / 140 0.2 / 1 / 2.5 8 / 18 / 35 No
Threshold rule: when multiple profiles are selected, Pingaro uses the most demanding threshold for each metric. In practice, it takes the lowest threshold among the selected profiles for RTT, loss, and jitter.

Overall Status Label

The bold overall status label appears under the total ping count. It summarizes connection quality without requiring users to read every chart.

Situation Message Pattern
No issues detected So far, the connection looks good.
One issue The connection has had a minor issue at 12:34.
Two or three issues of one severity The connection has had some noticeable issues (between 12:10-12:15).
Four or more issues of one severity The connection has had several serious issues (between 12:10-12:15).
Mixed severities The connection has had: a minor issue at 12:34, some noticeable issues (last at 12:20), several serious issues (between 12:10-12:15).

Source of Truth

  • Once aggregate samples exist, the status label is based on the aggregate charts: p95 RTT, loss, and visible jitter.
  • Before aggregate samples exist, the label falls back to the realtime RTT graph.
  • If the jitter graph is hidden, jitter is not considered for the overall status label.
  • The label background uses the same severity colors as the graph warning regions.

Layout Decisions

  • The first screen is the actual monitoring dashboard, not a landing page.
  • The left pane contains target groups, Internet uses, aggregation settings, and actions.
  • The right pane contains the realtime RTT chart and aggregate charts.
  • Chart headers, last values, plot area, axis labels, and warning regions are custom-painted inside Walk CustomWidget controls.
  • When jitter is hidden, its vertical space is redistributed to visible charts.
  • Loss and jitter charts can shrink when their data does not need full height, with the saved space redistributed to other charts.
  • Y-axis scaling uses bucketed maxima so the graph stays stable while still covering the visible data.

Defaults And Persistence

Area Decision Reason
Fresh target groups Group #1 is Gateway with target gateway. Group #2 is Internet. Local gateway quality is useful context before public Internet quality. Keeping the target as the special name gateway lets Pingaro resolve the current default gateway at runtime.
Special target names localhost resolves to 127.0.0.1. gateway resolves to the current default gateway IP address. Users can express stable intent in saved targets while Pingaro resolves machine-specific addresses when monitoring starts.
Internet targets 1.1.1.1, 1.1.1.2, 8.8.8.8, 8.8.4.4 Use multiple well-known targets so one host does not dominate the diagnosis.
Default selected uses email & browsing, audio calls, video calls, online gaming These cover common usage without immediately applying the strictest possible thresholds.
Default unselected uses remote desktop, Superhuman Gaming These are more specialized and can make the app more sensitive than most users expect.
Saved config Settings are stored in %AppData%\Pingaro\settings.json and reused on next launch. The filename should describe the file's purpose. settings.json is clearer than the old pingaro.json name inside a Pingaro-specific folder.
Legacy config cleanup If settings.json is missing but legacy pingaro.json exists, Pingaro reads it, writes settings.json, then deletes pingaro.json after the write succeeds. Users keep their settings during the rename, and stale duplicate state is removed immediately.

Build And Release Decisions

  • Repeatable Windows resources: go generate ./... runs github.com/akavel/rsrc@v0.10.2 to refresh rsrc.syso from the manifest and icon.
  • GUI subsystem build: release builds use go build -trimpath -ldflags="-H=windowsgui" -o pingaro.exe ..
  • One release asset: GitHub releases attach pingaro.exe. Users download and run that executable directly.
  • Unsigned app warning: release notes and README tell users how to run the unsigned executable only if it was downloaded from the official GitHub release page.

History And Compatibility

  • History files use JSON records with a version field.
  • Saved samples and aggregate points can be loaded back into the UI.
  • Legacy single-use configuration is normalized into the newer multi-use list.
  • Legacy pingaro.json settings are migrated to settings.json and cleaned up.
  • The old "low latency gaming" profile name is normalized to "Superhuman Gaming".
  • Loaded history can apply its saved config when present.

Decision Log Maintenance

This document is part of the product, not a one-time note. It should change whenever the product's intentional behavior or constraints change.

Rule Why It Exists
AI agents must update this file when changing product scope, naming, UX, target behavior, metrics, thresholds, colors, defaults, persistence, build/release behavior, or other design choices. The repository should preserve the reasoning behind choices, not only the implementation.
If a code change does not require a decision-log update, the agent should say so in its final response. This makes the check explicit and prevents silent drift.
Updates should be made in the same change as the behavior they document. Keeping documentation and implementation together makes review and future maintenance easier.
Document presentation should stay purposeful: use richer HTML only when it improves understanding, such as GUI mockups, severity swatches, threshold callouts, and tables for structured comparisons. The design log should be easy to read and maintain. Decoration such as sticky navigation or repeated status labels should not compete with the decisions themselves.
